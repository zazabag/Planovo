## 1. Анализ ScheduleKEMS

### 1.1 Общая архитектура

ScheduleKEMS представляет собой трёхмодульную систему управления расписанием учебного заведения. Каждый модуль ориентирован на свою целевую аудиторию и реализован с использованием различных технологических подходов:

| Модуль | Технология | Целевая аудитория |
|--------|-----------|-------------------|
| Public Viewer | React | Студенты и преподаватели (просмотр) |
| Admin Panel | Vanilla JS | Администраторы расписания |
| Teacher Cabinet | Vanilla JS | Преподаватели |

### 1.2 Хранение данных

Система работает **без серверного бэкенда** — все данные хранятся в `localStorage` браузера. Это решение имеет следующие последствия:

- **Преимущества:** мгновенный доступ к данным, отсутствие задержек сети, работа офлайн, простота развёртывания.
- **Ограничения:** данные привязаны к браузеру/устройству, нет синхронизации между пользователями, ограниченный объём хранилища (~5-10 МБ), невозможность совместной работы в реальном времени.

### 1.3 Модель данных: Schedule Cell

Основная единица данных — ячейка расписания (`Schedule Cell`). Модель включает следующие поля:

```
ScheduleCell {
  groupCode: string    // Код учебной группы (например, "ИС-21")
  dayKey: string       // День недели ("monday"–"friday" / "saturday")
  slotIndex: number    // Индекс временного слота (0-based)
  parity: string       // Чётность недели: "even" | "odd" | "all"
  subject: string      // Название дисциплины
  type: string         // Тип занятия: "lecture" | "seminar" | "lab" | "practice"
  teacher: string      // ФИО преподавателя
  room: string         // Аудитория
  combined: boolean    // Признак объединённой пары (несколько групп)
}
```

Поле `combined` позволяет отмечать пары, на которых присутствуют несколько групп одновременно (например, потоковые лекции). Поле `parity` поддерживает систему чередования по чётным/нечётным неделям.

### 1.4 Система чётности недель

Чётность недели определяется на основе **ISO-номера недели**:

- Если ISO-номер недели чётный → **чётная неделя** (`even`)
- Если ISO-номер недели нечётный → **нечётная неделя** (`odd`)

Это позволяет автоматически переключать отображаемое расписание в зависимости от текущей недели, что критически важно для дисциплин, проводящихся через неделю.

### 1.5 Модуль Public Viewer (React)

**Студенческий просмотр расписания** — основной модуль для конечных пользователей.

#### Десктоп-версия

1. Пользователь выбирает группу из селектора
2. Отображается недельная сетка расписания (таблица: дни × временные слоты)
3. Каждая ячейка содержит карточку пары с предметом, типом, преподавателем и аудиторией
4. Индикатор «Сейчас» подсвечивает текущую пару

#### Мобильная версия

1. Вместо сетки — дневной список (свайп между днями)
2. Карточки пар адаптированы под вертикальную прокрутку
3. Индикатор «Сейчас» и навигация по дням

#### Карточка пары (Pair Card)

Содержит:
- Название дисциплины (жирным)
- Тип занятия (цветовой значок: лекция, семинар, лабораторная)
- ФИО преподавателя
- Номер аудитории
- Признак чётности (если применимо)

### 1.6 Модуль Teacher Cabinet (Vanilla JS)

**Кабинет преподавателя** — интерфейс для управления доступностью и просмотра расписания.

#### Редактор доступности

- Интерфейс в виде сетки (дни × слоты)
- Преподаватель переключает слоты: «доступен» ↔ «недоступен»
- Недоступные слоты не могут быть назначены администратором

#### Просмотр расписания

- Только чтение (read-only)
- Отображение всех пар преподавателя на текущую неделю
- Фильтрация по чётности

### 1.7 Модуль Admin Panel (Vanilla JS)

**Панель администратора** — полноценный конструктор расписания.

#### Конструктор расписания

- Селекторы: предмет, преподаватель, аудитория
- Drag-and-drop или клик-назначение в сетку расписания
- Поддержка назначения на конкретную чётность недели

#### Система обнаружения конфликтов

При назначении пары система проверяет три типа конфликтов:

| Тип конфликта | Проверка |
|---------------|----------|
| Групповой | Пересечение пар для одной группы в одном слоте |
| Преподавательский | Пересечение пар для одного преподавателя в одном слоте |
| Аудиторный | Пересечение пар в одной аудитории в одном слоте |

При обнаружении конфликта отображается предупреждение с указанием типа и участников.

#### Публикация расписания

После завершения редактирования администратор публикует расписание, что делает его доступным для просмотра в Public Viewer.

### 1.8 Ключевые особенности ScheduleKEMS

1. **Детекция конфликтов** — тройная проверка (группа/преподаватель/аудитория) при каждом назначении
2. **Объединённые пары** — поддержка потоковых лекций с несколькими группами
3. **Детерминированные мок-данные** — генерация тестовых данных с фиксированным seed для воспроизводимости
4. **Чётность недель** — автоматическое переключение расписания по ISO-номеру недели
5. **Адаптивный интерфейс** — разные представления для десктопа и мобильных устройств

## 2. Анализ дизайна Planovo

### 2.1 Цветовая палитра

Дизайн-система Planovo построена на индиго-фиолетовой палитре с акцентными цветами:

| Роль | Цвет | HEX |
|------|------|-----|
| Primary | Индиго | `#6366f1` |
| Secondary | Фиолетовый | `#8b5cf6` |
| Accent | Бирюзовый | `#06b6d4` |
| Dark | Тёмно-синий | `#0f172a` |

#### Основной градиент

```
linear-gradient(135deg, #6366f1 0%, #8b5cf6 50%, #a855f7 100%)
```

Градиент используется для:
- Фона hero-секций
- Кнопок призыва к действию (CTA)
- Декоративных элементов и иконок
- Заголовков и акцентного текста

### 2.2 Типографика

**Шрифт:** Inter (Google Fonts)

| Начертание | Вес | Применение |
|------------|-----|------------|
| Light | 300 | Дополнительный текст, подписи |
| Regular | 400 | Основной текст |
| Medium | 500 | Подзаголовки, кнопки |
| SemiBold | 600 | Заголовки карточек |
| Bold | 700 | Секционные заголовки |
| ExtraBold | 800 | Главные заголовки |
| Black | 900 | Hero-заголовки |

### 2.3 Карточная система UI

Основа интерфейса — **карточки** с единообразным стилем:

```css
.card {
  border-radius: 16px;
  border: 1px solid #e5e7eb;
  background: #ffffff;
  padding: 24px;
  transition: all 0.3s ease;
}
```

#### Hover-эффект

При наведении карточка «приподнимается» и получает усиленную тень:

```css
.card:hover {
  transform: translateY(-4px);
  box-shadow: 0 20px 25px -5px rgba(0,0,0,0.1),
              0 10px 10px -5px rgba(0,0,0,0.04);
  border-color: #818cf8;
}
```

Смена цвета рамки на `#818cf8` (светлый индиго) создаёт визуальную связь с основной палитрой.

### 2.4 Навигационная панель: Glass Morphism

Навбар реализован в стиле **стеклянного морфизма**:

```css
.navbar {
  background: rgba(255, 255, 255, 0.8);
  backdrop-filter: blur(20px);
  -webkit-backdrop-filter: blur(20px);
  border-bottom: 1px solid rgba(229, 231, 235, 0.5);
}
```

Эффект размытия фона (`blur(20px)`) при полупрозрачном фоне создаёт ощущение глубины и современности, одновременно обеспечивая читаемость навигации при прокрутке.

### 2.5 Система теней

Planovo использует градуированную систему теней:

| Уровень | CSS-класс | Значение |
|---------|-----------|----------|
| SM | `.shadow-sm` | `0 1px 2px rgba(0,0,0,0.05)` |
| MD | `.shadow-md` | `0 4px 6px -1px rgba(0,0,0,0.1)` |
| LG | `.shadow-lg` | `0 10px 15px -3px rgba(0,0,0,0.1)` |
| XL | `.shadow-xl` | `0 20px 25px -5px rgba(0,0,0,0.1)` |

Тени усиливаются при hover и в активных состояниях, создавая иерархию глубины.

### 2.6 Три ниши Planovo

Система позиционируется в трёх рыночных нишах с уникальными цветовыми решениями:

#### Образование (Education)

- **Палитра:** Индиго-фиолетовая (`#6366f1` → `#8b5cf6`)
- **Иконки:** Академическая тематика (шапка, книга, доска)
- **Применение:** Школы, колледжи, университеты

#### Спорт (Sports)

- **Палитра:** Янтарно-красная (`#f59e0b` → `#ef4444`)
- **Иконки:** Спортивная тематика (мяч, таймер, трофей)
- **Применение:** Спортивные секции, фитнес-клубы, тренировки

#### Клубы (Clubs)

- **Палитра:** Изумрудно-бирюзовая (`#10b981` → `#06b6d4`)
- **Иконки:** Креативная тематика (палитра, музыка, код)
- **Применение:** Кружки, творческие студии, IT-клубы

### 2.7 Теги секций (Section Tags)

Секционные метки реализованы как «таблетки» (pills):

```css
.section-tag {
  background: rgba(99, 102, 241, 0.1);
  color: #6366f1;
  padding: 4px 12px;
  border-radius: 9999px;
  font-size: 0.875rem;
  font-weight: 500;
}
```

Полупрозрачный фон `rgba(99,102,241,0.1)` создаёт мягкий цветовой акцент без визуального перегруза.

### 2.8 Анимации появления (Scroll-Reveal)

Элементы появляются при прокрутке с **каскадными задержками**:

```
Элемент 1: delay 0ms
Элемент 2: delay 100ms
Элемент 3: delay 200ms
Элемент 4: delay 300ms
...
```

Техническая реализация:
- Intersection Observer отслеживает попадание элементов в viewport
- Каждый элемент получает `transition-delay` на основе индекса
- Анимация: `opacity: 0 → 1`, `transform: translateY(30px) → translateY(0)`
- Длительность: 0.6s, easing: `cubic-bezier(0.4, 0, 0.2, 1)`

### 2.9 Иконки

**Библиотека:** Font Awesome

Иконки используются для:
- Навигационных элементов
- Индикаторов в карточках
- Декоративных элементов секций
- Кнопок действий

### 2.10 Ключевые принципы дизайна Planovo

1. **Единообразие** — все компоненты следуют единой дизайн-системе
2. **Глубина** — система теней и hover-эффекты создают иерархию
3. **Плавность** — все переходы анимированы (0.3s ease)
4. **Адаптивность** — мобильная и десктопная версии с разными компоновками
5. **Цветовое кодирование** — каждая ниша имеет уникальную палитру

## 3. Анализ Planora (автогенерация расписаний)

### 3.1 Общее описание

Planora — модуль автоматической генерации расписания на основе **Google OR-Tools CP-SAT** (Constraint Programming — SATisfiability). Это система решения задач удовлетворения ограничений, которая позволяет находить оптимальные или допустимые расписания с учётом множества жёстких и мягких ограничений.

### 3.2 Входные данные

#### Преподаватели (Teachers)

```
Teacher {
  id: string
  name: string
  maxDailyLoad: number     // Макс. количество пар в день
  maxWeeklyLoad: number    // Макс. количество пар в неделю
  unavailableSlots: []     // Недоступные временные слоты
}
```

#### Дисциплины (Subjects)

```
Subject {
  id: string
  name: string
  type: "lecture" | "seminar" | "lab" | "practice"
  classDuration: number    // Длительность в слотах (1 или 2 для лабораторных)
  daysPerWeek: number      // Количество занятий в неделю
  teacherId: string        // Закреплённый преподаватель
  groupId: string          // Учебная группа
  roomRequirement: string  // Требование к аудитории (опционально)
}
```

#### Аудитории и лаборатории (Rooms/Labs)

```
Room {
  id: string
  name: string
  capacity: number
  type: "standard" | "lab" | "gym" | "special"
  equipment: string[]      // Доступное оборудование
}
```

#### Филиалы (Branches)

```
Branch {
  id: string
  name: string
  rooms: Room[]
  groups: Group[]
}
```

#### Пользовательские правила (Custom Rules)

Пользователь может задавать произвольные ограничения:
- «Преподаватель X не работает по пятницам»
- «Дисциплина Y должна быть в первой половине дня»
- «Группа Z не должна иметь окна более 1 пары»

### 3.3 Жёсткие ограничения (Hard Constraints)

Жёсткие ограничения **должны** быть выполнены. Их нарушение делает расписание недопустимым.

| # | Ограничение | Описание |
|---|-------------|----------|
| H1 | Отсутствие пересечений преподавателей | Один преподаватель не может вести две пары одновременно |
| H2 | Отсутствие пересечений аудиторий | Одна аудитория не может быть занята двумя парами одновременно |
| H3 | Отсутствие пересечений групп | Одна группа не может иметь две пары одновременно |
| H4 | Максимальная дневная нагрузка | Количество пар у преподавателя в день не превышает maxDailyLoad |
| H5 | Максимальная недельная нагрузка | Количество пар у преподавателя в неделю не превышает maxWeeklyLoad |
| H6 | Перерывы | Учитывание обязательных перерывов (обед, перемены) |
| H7 | Фиксированные назначения | Принудительное размещение пары в конкретный слот |
| H8 | Последовательные периоды для лабораторных | Лабораторные занятия (2 слота) размещаются в смежных слотах |

### 3.4 Мягкие ограничения (Soft Constraints)

Мягкие ограничения **желательны**, но могут быть нарушены. Нарушение ухудшает целевую функцию, но не делает расписание недопустимым.

| # | Ограничение | Описание |
|---|-------------|----------|
| S1 | Балансировка нагрузки | Равномерное распределение пар по дням для преподавателей и групп |
| S2 | Целевые показатели сессий | Стремление к определённому количеству пар в день для групп |

### 3.5 Режимы решения

#### Строгий режим (Strict)

- Все жёсткие и мягкие ограничения активны
- Решение ищется с максимальным соблюдением всех ограничений
- Может не найти решение при противоречивых ограничениях
- Время решения: от нескольких секунд до нескольких минут

#### Частичный режим (Partial / Relaxed)

- Мягкие ограничения ослаблены или отключены
- Некоторые жёсткие ограничения могут быть переведены в мягкие
- Находит решение быстрее
- Гарантированно получает расписание (возможно, с компромиссами)

### 3.6 Выходные данные

#### Сетка расписания (Timetable Grid)

Результат генерации — полная сетка расписания с заполненными ячейками:

```
TimetableResult {
  schedule: ScheduleCell[]      // Массив ячеек расписания
  diagnostics: Diagnostic[]     // Диагностика пустых ячеек
  stats: GenerationStats        // Статистика генерации
}
```

#### Диагностика

Для каждой пустой ячейки система предоставляет объяснение, почему она не была заполнена:

```
Diagnostic {
  dayKey: string
  slotIndex: number
  groupCode: string
  reason: string               // "Нет доступного преподавателя"
                               // "Аудитория занята"
                               // "Превышена дневная нагрузка"
}
```

Это критически важная функция — она позволяет администратору понять, почему расписание не идеально, и скорректировать входные данные.

### 3.7 Кинематографическая загрузочная анимация

Во время генерации расписания (которая может занимать от секунд до минут) отображается **кинематографическая загрузочная анимация**:

- Визуализация процесса решения (подсчёт ограничений, итерации)
- Прогресс-бар с этапами
- Отображение текущего состояния решения
- Переход к результату с анимацией «раскрытия»

### 3.8 Экспорт расписания

Сгенерированное расписание может быть экспортировано в следующие форматы:

| Формат | Библиотека | Особенности |
|--------|-----------|-------------|
| PDF | jsPDF | Печатная версия с форматированием |
| Excel | ExcelJS | Редактируемая таблица с листами по группам |
| Word | docx.js | Документ с таблицами |
| Print | window.print() | Системный диалог печати |

### 3.9 Ключевые особенности Planora

1. **CP-SAT решатель** — индустриальный стандарт для задач составления расписаний
2. **Иерархия ограничений** — жёсткие vs. мягкие, строгий vs. частичный режимы
3. **Диагностика** — объяснение причин пустых ячеек
4. **Настраиваемость** — пользовательские правила и ограничения
5. **Мультиформатный экспорт** — PDF, Excel, Word, печать
6. **Зрелищная генерация** — кинематографическая анимация процесса

## 4. Интеграция и план реализации демо-версии Planovo

### 4.1 Общая концепция

Демо-версия Planovo объединяет три проанализированных компонента в единый продукт:

```
┌─────────────────────────────────────────────────┐
│                    Planovo                       │
│                                                  │
│  ┌──────────────┐  ┌──────────┐  ┌────────────┐│
│  │ ScheduleKEMS │  │ Дизайн   │  │  Planora   ││
│  │  (Данные и   │  │ Planovo  │  │ (Генерация)││
│  │  UI-модели)  │  │ (Визуал) │  │            ││
```

### 4.2 Принципы интеграции

#### Дизайн → UI

- Все компоненты используют дизайн-систему Planovo (цвета, типографика, карточки)
- Glass morphism навбар и scroll-reveal анимации применяются ко всем модулям
- Цветовое кодирование ниш адаптируется для типов занятий:
  - Лекции → индиго
  - Семинары → бирюзовый
  - Лабораторные → фиолетовый
  - Практика → изумрудный

#### ScheduleKEMS → Модели данных

- Модель `ScheduleCell` используется как общий формат данных
- Система чётности недель наследуется без изменений
- Конфликт-детекция ScheduleKEMS дополняет ограничения Planora

#### Planora → Генерация

- Planora вызывается из Admin Panel как «автогенерация»
- Результат генерации конвертируется в массив `ScheduleCell`
- Диагностика Planora отображается в интерфейсе администратора
- Экспорт Planora доступен из всех модулей

### 4.3 План реализации

#### Этап 1: Фундамент (Неделя 1)

1. **Настройка проекта**
   - Инициализация React-приложения с TypeScript
   - Подключение Tailwind CSS с дизайн-токенами Planovo
   - Настройка маршрутизации (React Router)

2. **Дизайн-система**
   - Создание компонентной библиотеки: кнопки, карточки, инпуты, теги
   - Реализация glass morphism навбара
   - Настройка типографики Inter

3. **Модели данных**
   - TypeScript-интерфейсы для `ScheduleCell`, `Teacher`, `Subject`, `Room`
   - Утилиты для работы с localStorage
   - Сервис чётности недель

#### Этап 2: Модуль просмотра (Неделя 2)

1. **Public Viewer**
   - Селектор групп
   - Недельная сетка расписания (десктоп)
   - Дневной список (мобильный)
   - Карточки пар с цветовым кодированием типов
   - Индикатор «Сейчас»
   - Scroll-reveal анимации

2. **Teacher Cabinet**
   - Редактор доступности (toggle-сетка)
   - Просмотр расписания преподавателя (read-only)

#### Этап 3: Модуль администратора (Неделя 3)

1. **Admin Panel — Конструктор**
   - Сетка расписания с drag-and-drop
   - Селекторы: предмет, преподаватель, аудитория
   - Назначение с учётом чётности недели
   - Обнаружение конфликтов в реальном времени

2. **Admin Panel — Публикация**
   - Предпросмотр перед публикацией
   - Механизм публикации (localStorage → Public Viewer)
   - Отмена/откат изменений

#### Этап 4: Интеграция Planora (Неделя 4)

1. **Модуль автогенерации**
   - Интеграция Google OR-Tools CP-SAT (через WASM или серверный API)
   - Интерфейс настройки ограничений
   - Кинематографическая загрузочная анимация
   - Отображение результатов и диагностики

2. **Экспорт**
   - PDF-экспорт (jsPDF)
   - Excel-экспорт (ExcelJS)
   - Word-экспорт (docx.js)
   - Печать (window.print)

#### Этап 5: Полировка и демо (Неделя 5)

1. **Оптимизация**
   - Адаптивность на всех разрешениях
   - Производительность рендеринга больших расписаний
   - Обработка крайних случаев

2. **Демо-данные**
   - Детерминированные мок-данные ScheduleKEMS
   - Реалистичное расписание на неделю (5-6 групп, 10-15 преподавателей)
   - Примеры конфликтов и их разрешения

3. **Финальное тестирование**
   - Кросс-браузерное тестирование
   - Мобильное тестирование
   - Проверка генерации расписания Planora

### 4.5 Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| OR-Tools WASM слишком тяжёлый | Среднее | Высокое | Серверный API как fallback |
| localStorage переполнен | Низкое | Среднее | Сжатие данных, лимит расписаний |
| Мобильная сетка неработоспособна | Низкое | Среднее | Переключение на дневной список |
| Долгая генерация расписания | Среднее | Низкое | Частичный режим + анимация |
| Конфликты дизайна между нишами | Низкое | Низкое | Чёткая изоляция цветовых тем |

### 4.6 Ожидаемый результат

Демо-версия Planovo будет представлять собой полнофункциональное веб-приложение, демонстрирующее:

1. **Просмотр расписания** — красивый и интуитивный интерфейс для студентов и преподавателей
2. **Управление расписанием** — конструктор с обнаружением конфликтов для администраторов
3. **Автогенерация** — интеллектуальное составление расписания с помощью OR-Tools
4. **Единый визуальный стиль** — дизайн-система Planovo, объединяющая все модули
