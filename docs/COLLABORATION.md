# Совместная работа над Planovo

Документ для двух (и более) разработчиков, чтобы не мешать друг другу и не терять контекст.  
**Главная инструкция агента:** `.cursor/rules/git-workflow.mdc` — всегда ссылается на этот файл.

## Золотые правила

1. **Сначала `git pull`** — перед любой новой веткой: `git checkout main && git pull origin main`.
2. **Одна задача = одна ветка = один PR** — не смешивать несвязанные правки.
3. **Заявляй задачу** — запиши в [TASKS.md](./TASKS.md) до начала кода (статус `in-progress`).
4. **Маленькие PR** — проще ревьюить другу и меньше конфликтов.
5. **Не коммить в `main` напрямую** — только через PR и merge.
6. **Документируй сразу** — `UPDATE-LOG.md`, `WORKLOG.md`, при необходимости README.

## Именование веток

```
<тип>/<github-user>/<краткое-описание>
```

| Тип | Когда |
|-----|-------|
| `feature/` | новая функция |
| `fix/` | исправление бага |
| `docs/` | только документация |

Примеры:
- `feature/zazabag/landing-hero`
- `feature/Owl-14/education-autogen`
- `fix/zazabag/sports-jsx-bracket`

## Реестр задач — [TASKS.md](./TASKS.md)

Перед стартом работы агент **обязан**:

1. Открыть `TASKS.md`.
2. Найти свободную задачу или создать новую строку.
3. Указать владельца, ветку, статус `in-progress`.
4. После merge — обновить статус на `done` и ссылку на PR.

Так друг видит, что уже занято, и не дублирует работу.

## Карта репозитория (где чаще конфликты)

| Путь | Содержимое | Совет |
|------|------------|-------|
| `index.html`, `404.html` | Статический лендинг | один PR на секцию лендинга |
| `demo/education/` | Исходники Next.js демо | не трогать чужую активную ветку |
| `docs/` | Документация | merge-конфликты решаются вручную, не перетирать чужие записи |
| `_next/` | Сборка | **не редактировать вручную** — только из build-процесса |
| `.cursor/rules/` | Правила Cursor | согласовывать изменения workflow |

## Цикл работы (чеклист)

```
[ ] git pull origin main
[ ] Задача заявлена в TASKS.md
[ ] Создана ветка feature/<user>/...
[ ] Код + документация
[ ] Commit + push
[ ] PR создан (gh pr create), друг указан ревьюером при возможности
[ ] Ревью / правки
[ ] Merge в main
[ ] TASKS.md → done, WORKLOG.md обновлён
[ ] git checkout main && git pull
```

## Pull Request — что писать другу

В описании PR обязательно:

- **Кто:** GitHub-ник автора
- **Задача:** ID из `TASKS.md`
- **Затронутые пути:** список папок/файлов
- **Как проверить:** шаги для ручной проверки
- **Документация:** какие md-файлы обновлены

Пример заголовка: `feat(education): автогенерация расписания [Owl-14, TASK-4]`

## Конфликты

1. Не делать `git push --force` в общие ветки.
2. При конфликте в `docs/`: **сохранить обе записи** (оба changelog / оба task-блока в WORKLOG).
3. При конфликте в коде: автор PR разрешает локально, просит повторное ревью.
4. Если два PR трогают один файл — merge по очереди; второй делает rebase на `main`.
